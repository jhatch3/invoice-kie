**Stack:** Vue 3 + Vite + Pinia + vue-router + TypeScript (existing `src/frontend/` scaffold)

## Goal

A single-page frontend mockup for invoice-kie that lets a user **see the full extraction
pipeline end-to-end** before any backend exists:

1. Download a bundled test invoice PDF.
2. Upload that PDF back into the app.
3. Run it — the app calls a **stubbed** extraction API and populates a results table with the
   extracted fields (total, tax, subtotal, date, invoice number).

The table starts blank. "Running" a file is wired through a real data layer so the mock can be
swapped for the real LayoutLMv3 backend later with no UI changes.

## Non-goals

- No real ML/OCR pipeline (the backend does not exist yet).
- No authentication, persistence, or multi-user features.
- No production deployment config.

## Architecture

Single route (`/`) rendering `HomeView.vue`. All UI is decomposed into small
single-purpose components, **each in its own file** under `src/frontend/src/components/`.

### Data layer (stubbed, backend-ready)

- **TanStack Query (Vue Query)** provides a mutation `useExtraction()` that POSTs the PDF
  (multipart/form-data) to `POST /api/extract`.
- **MSW (Mock Service Worker)** intercepts `/api/extract` in the browser and returns a mock
  `ExtractionResult` after a ~1.2s delay. Results are matched by uploaded filename to the
  bundled samples; unknown files get a generic result.
- Swapping to the real backend later = remove the MSW handler and point the client at the real
  URL. No component changes.

### State

- **Query cache** holds the extraction result for each run (server-owned data).
- **Pinia store (`stores/runs.ts`)** holds the client-owned list of runs (which files the user
  added and each row's UI status). This is genuinely client state, not fetched data.

## Components (one file each, `src/frontend/src/components/`)

| Component | Responsibility | Key states |
|-----------|----------------|-----------|
| `AppHeader.vue` | App title + one-line description | — |
| `PipelineDiagram.vue` | Visualizes `PDF → OCR → LayoutLMv3 → JSON` so the user sees the pipeline | static; optional active-step highlight during a run |
| `SampleDownload.vue` | Buttons to download the two bundled sample invoices | — |
| `UploadDropzone.vue` | Drag-drop + file picker; `.pdf`-only validation; keyboard accessible | idle / dragover / invalid-file |
| `RunButton.vue` | Triggers the extraction mutation for pending file(s) | idle / pending / disabled |
| `ExtractionTable.vue` | Results table | empty / loading (skeleton) / populated |
| `StatusBadge.vue` | Per-row status pill | idle / running / done / error |

### Supporting files (each its own file)

- `views/HomeView.vue` — composes the components; owns page layout.
- `composables/useExtraction.ts` — Vue Query mutation wrapping the API client.
- `api/extractClient.ts` — `fetch` wrapper posting multipart to `/api/extract`; typed return.
- `mocks/handlers.ts` — MSW handler for `/api/extract` with per-filename mock results + delay.
- `mocks/browser.ts` — MSW worker setup; started in `main.ts` in dev only.
- `stores/runs.ts` — Pinia setup store: list of runs, add/update/reset actions.
- `types/invoice.ts` — `ExtractionResult`, `RunRow`, `RunStatus` types (the boundary contract).
- `router/index.ts` — add the `/` → `HomeView` route (currently empty `routes: []`).

## Data model (`types/invoice.ts`)

```ts
type RunStatus = 'idle' | 'running' | 'done' | 'error'

interface ExtractionResult {
  invoiceNumber: string | null
  date: string | null          // ISO yyyy-mm-dd
  subtotal: number | null      // major units
  tax: number | null
  total: number | null
  currency: string             // e.g. 'USD'
  confidence: number           // 0..1
}

interface RunRow {
  id: string                   // stable client id
  fileName: string
  status: RunStatus
  result: ExtractionResult | null
  error: string | null
}
```

## Table columns

File · Invoice # · Date · Subtotal · Tax · Total · Confidence · Status

- Empty state: friendly message prompting the user to download a sample and upload it.
- Currency values formatted with `Intl.NumberFormat`; dates with `Intl.DateTimeFormat`.
- `null` fields render as an em-dash `—`.

## Sample PDFs

- **Two** sample invoice PDFs with visibly different layouts (e.g. `acme-invoice.pdf`,
  `globex-invoice.pdf`), generated by a small build-time Node script (`scripts/gen-samples.mjs`
  using `pdf-lib` as a devDependency).
- Canonical copies live in `src/data/`; the script also writes them to
  `src/frontend/public/samples/` so Vite can serve them for download.
- Each sample's known-correct fields drive its MSW mock result, so download → upload → run is
  internally consistent.

## Data flow

```
SampleDownload  ──download──▶  browser saves /samples/acme-invoice.pdf
        │
        ▼ (user uploads it back)
UploadDropzone  ──valid .pdf──▶  runs store: add RunRow (status: idle)
        │
        ▼ (click Run)
RunButton ─▶ useExtraction.mutate(file) ─▶ POST /api/extract
        │                                       │
        │                                   MSW intercepts (dev)
        ▼                                       ▼
runs store: status 'running'            returns ExtractionResult (~1.2s)
        │                                       │
        ▼◀──────────────────────────────────────┘
runs store: status 'done' + result   (or 'error' on failure)
        │
        ▼
ExtractionTable renders the row
```

## Error handling

- Non-PDF upload → rejected in `UploadDropzone` with an inline, screen-reader-announced message.
- Mutation failure → row status `error`, error text in the Status cell, a retry affordance.
- MSW can be toggled to return a 500 for one sample to demonstrate the error path.

## Quality bar (frontend + vue skills)

- **a11y:** semantic `<table>` with `<th scope>`; dropzone is a labelled `<input type=file>`
  styled as a dropzone (keyboard operable, visible focus); status changes announced via an
  `aria-live="polite"` region; contrast AA; respects `prefers-reduced-motion`.
- **States:** loading (skeleton rows), error, and empty are all designed, not just success.
- **TypeScript:** strict; boundary types in `types/invoice.ts`; no `any`.
- **Styling:** design tokens (CSS custom properties) for color/spacing; light + dark theme;
  scoped SFC styles; mobile-first responsive; table scrolls horizontally in its own container.
- **Reactivity:** `<script setup lang="ts">`, typed `defineProps`/`defineEmits`, `computed`
  for derived values, Pinia setup store with `storeToRefs`.

## Testing

- **Vitest + @vue/test-utils** — component behavior:
  - `ExtractionTable`: empty / loading / populated rendering; `null` → `—`; currency/date format.
  - `UploadDropzone`: accepts `.pdf`, rejects other types with an announced error.
  - `StatusBadge`: renders the right label/variant per status.
- **MSW in tests** — `useExtraction` resolves to a mock result; error path sets row error.
- **Cypress e2e** — happy path: upload a fixture PDF → Run → assert row populates with expected
  fields. (Download step covered manually; e2e uploads a fixture directly.)

## Open decisions (resolved)

- Framework: **Vue 3** (existing scaffold). *(React was briefly considered and dropped.)*
- Run behavior: **stubbed real API** via Vue Query + MSW.
- Samples: **two** bundled PDFs, sourced from `src/data`, served from `public/samples`.
